# 智研舱方案创新点说明（答辩备忘）

## 1. 这套方案的核心创新点

### 1.1 用项目主线取代聊天主线

传统知识库产品常把用户带进一个聊天窗口，再在窗口里临时找资料。

智研舱把“项目”放在最上层。资料、问答、对比、记忆、输出、运行都挂在同一个项目空间里。这样用户面对的不是一次性的问答，而是一条可以持续推进的项目工作流。

### 1.2 用证据卡片约束回答

回答不是只给结论。

每次问答都尽量返回证据卡片，卡片至少带：

- `source_name`
- `citation_text`
- `internal_ref`

当证据不足时，系统应该明确提示证据不足，而不是继续给一个看起来完整但无法回溯的说法。

### 1.3 把 compare 做成结构化分析

很多产品只会把两份文档总结成一段散文。

智研舱把 compare 结果拆成：

- 已覆盖项
- 缺失项
- 风险项
- 待人工确认项

这样 compare 才能服务真实项目推进，而不是只服务展示。

### 1.4 用长期记忆治理稳定事实

项目里会有很多反复出现的稳定事实。

智研舱把这些事实沉淀成长期记忆，并要求每条记忆带 `source_refs`。这样后续问答、对比、输出就不需要每次都重新整理整套上下文。

### 1.5 让运行轨迹可见

项目任务往往不是瞬时完成的。

智研舱把 run、evidence、memory、artifact 之间的关系记录下来。用户能看到任务有没有成功、失败在哪里、是否需要重试。

## 2. 如果评委问“创新点是什么”

建议回答口径：

1. 不是把聊天框做大，而是把项目作为一级对象
2. 不是只生成答案，而是给证据、对比、记忆和产物
3. 不是只展示结果，而是保留运行轨迹和失败状态

## 3. 最适合现场强调的三个创新点

如果现场时间不够，优先强调这三点：

1. 项目空间是一条完整主线
2. 证据卡片让回答可追溯
3. 长期记忆和运行轨迹让系统可治理

## 4. 一句话版本

智研舱的创新点，不是“更会聊天”，而是把项目级证据、分析、记忆和交付串成一条能落地的工作流。
