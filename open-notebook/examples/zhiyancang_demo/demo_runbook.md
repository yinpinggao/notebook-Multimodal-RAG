# 3 分钟演示与兜底手册

## 1. 标准演示顺序

推荐顺序：

1. 总览页
2. 证据问答
3. compare
4. memory
5. outputs
6. runs

其中前五步要体现完整闭环，第六步用来说明状态可见和失败可追踪。

## 2. 如果只剩一分钟，哪些页面必须保住

必须保住三页：

1. `overview`
2. `evidence`
3. `outputs`

原因如下：

- `overview` 负责先把项目场景和资料范围讲清楚
- `evidence` 负责证明回答来自资料，不是模型临场编的
- `outputs` 负责证明系统最后能交付答辩相关产物

如果还有余量，再补 `compare`。`memory` 和 `runs` 可以作为追问时的加分项展开。

## 3. 现场最稳的提问顺序

第一问建议：

- 当前方案覆盖了哪些评分项？

第二问建议：

- 这个产品为什么不是普通聊天式知识库？

第三问建议：

- 如果现场只剩一分钟，哪些页面必须保住？

## 4. 兜底说法

如果某个异步任务还没完成，现场统一口径：

> 这一步的状态已经被记录，当前先展示已完成的证据、对比或产物，再说明失败原因和重试入口。

## 5. 哪些页面适合回答不同问题

### 5.1 问“创新点是什么”

优先回到：

- `overview`
- `evidence`

因为创新点需要同时讲产品主线和证据约束。

### 5.2 问“为什么可信”

优先回到：

- `evidence`
- `compare`

### 5.3 问“为什么不是聊天机器人”

优先回到：

- `overview`
- `memory`
- `outputs`

## 6. 最小通过口径

如果只能保住最小闭环，至少要做到：

1. 一个回答带引用
2. 一个结构化 compare 结果
3. 一个可展示产物
