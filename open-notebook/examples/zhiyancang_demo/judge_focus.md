# 评委追问清单与风险提醒

## 1. 评委高概率追问

### 1.1 你们如何证明答案不是编的？

建议回答：

- 所有回答都尽量绑定证据卡片
- 证据里至少展示来源文件、引用文本和内部引用标识
- 当证据不足时，系统应该明确提示证据不足

### 1.2 compare 结果为什么可信？

建议回答：

- compare 不是只给一段总结
- 结果按覆盖项、缺失项、风险项、待确认项拆开
- 每一类都应该能回到来源文档

### 1.3 什么记忆可以写进去？

建议回答：

- 只有稳定事实和用户确认内容
- 未确认结论不能直接进入长期记忆
- 每条长期记忆都必须带 `source_refs`

### 1.4 为什么要单独做 runs 和 jobs？

建议回答：

- 因为项目任务常常是异步的
- 用户需要知道任务是在排队、运行、成功还是失败
- 失败后需要清楚提示，并支持重试

## 2. 演示时最容易掉分的点

1. 用“notebook”做主叙事，没有切到项目空间
2. 回答里只有结论，没有展示引用
3. compare 结果只是散文，没有结构
4. 记忆没有来源
5. 产物和 run 之间无法对应

## 3. 建议现场顺序

1. 总览页先说项目主线
2. 证据页证明回答可追溯
3. compare 页证明系统会分析，不只会聊天
4. memory 页证明系统会记住项目
5. outputs 页证明系统能交付成果

## 4. 兜底方案

如果现场模型较慢或异步任务未完成，优先保住这三件事：

1. 已有 demo 项目能打开
2. 已有 compare 或输出结果能展示
3. 失败状态和重试入口能说明白
