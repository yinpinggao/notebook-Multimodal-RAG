# 智研舱参赛方案说明（演示版）

## 1. 项目名称

智研舱 / ZhiyanCang

英文副标题：Project Memory Copilot

## 2. 一句话定位

一个面向科研、竞赛和项目申报场景的多模态项目工作台。它围绕单个项目组织资料、证据、记忆、对比和输出，不把用户困在单轮聊天里。

## 3. 方案目标

针对比赛任务书里的五个核心环节，方案目标如下：

1. 用项目空间取代零散 notebook 入口
2. 用证据卡片约束问答质量
3. 用结构化 compare 支持规则与方案对齐
4. 用可审查的项目记忆沉淀稳定事实
5. 用输出页生成答辩材料和总结材料

## 4. 系统主线

### 4.1 项目空间

- 以项目为一级入口
- 项目内统一呈现总览、证据、对比、记忆、输出、运行

### 4.2 证据问答

- 用户问题进入统一问答入口
- 系统根据问题类型选择文本检索或多模态检索
- 回答附带证据卡片

### 4.3 对比分析

- 支持规则文档 vs 方案文档
- 输出覆盖项、缺失项、风险项、待确认项

### 4.4 长期记忆

- 只沉淀稳定事实和用户确认内容
- 每条记忆要求带 `source_refs`
- 记忆状态分为待确认、已接受、已冻结

### 4.5 成果输出

当前演示聚焦三类产物：

- 答辩提纲
- 评委问题卡片
- 差异摘要

## 5. 对任务书要求的覆盖情况

| 任务书要求 | 当前方案 |
|---|---|
| 项目空间 | 已覆盖，按项目工作流组织入口 |
| 证据问答 | 已覆盖，回答绑定证据卡片 |
| 多文档对比 | 已覆盖，支持结构化差异输出 |
| 长期记忆 | 已覆盖，要求 `source_refs` |
| 成果输出 | 已覆盖，聚焦答辩与总结产物 |
| 运行轨迹 | 已覆盖，记录 run、证据、记忆和产物 |

## 6. 当前风险与边界

1. 复杂多文档 compare 仍以两两对比为主
2. 记忆治理仍需要人工确认闭环
3. 演示环境里异步任务速度受模型和硬件影响
4. 管理页先做最小可用，不追求完整后台

## 7. 推荐演示顺序

1. 打开 demo 项目总览
2. 询问“当前方案覆盖了哪些评分项”
3. 打开 compare，看任务书与方案说明差异
4. 打开 memory，看稳定事实是否带来源
5. 打开 outputs，生成答辩提纲或问题卡片

## 8. 适合评委追问的证据点

### 8.1 为什么是项目工作台

因为用户需要处理的是持续演进的项目上下文，而不是一次性的问答。项目空间能把资料、问题、结论、记忆和产物串在一起。

### 8.2 如何降低幻觉

通过证据卡片和引用字段约束回答。当证据不足时，系统应直接提示证据不足。

### 8.3 为什么需要长期记忆

因为项目中的稳定事实会在多轮问答、对比和输出里反复使用。如果每次都重新检索，成本高且一致性差。

## 9. 演示版预期结果

- 3 分钟内完成完整闭环
- 至少 1 个回答展示证据
- 至少 1 个 compare 结果展示缺失项
- 至少 1 条长期记忆展示来源
- 至少 1 个输出产物可用于答辩
